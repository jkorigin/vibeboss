## Phase 0 — Research

SKIPPED. Hook JSON schema is fully specified in the task brief. No unknown APIs.

## Phase 1 — Build

1. `mkdir -p hq/.claude/hooks`
2. Write `hq/.claude/settings.json` — SessionStart hook, matchers: startup + resume
3. Write `hq/.claude/hooks/boot.sh` — reads STATE.md, runlog/, inbox/, projects/, crew.yml;
   emits JSON with `hookSpecificOutput.additionalContext`
4. `chmod +x hq/.claude/hooks/boot.sh`
5. Run `shellcheck` (if available) or manually verify no syntax errors

## Phase 2 — Test

Run: `hq/.claude/hooks/boot.sh > /tmp/boot-smoke.json`

Verify with:
```bash
python3 -c "
import json,sys
d=json.load(open('/tmp/boot-smoke.json'))
ac=d['hookSpecificOutput']['additionalContext']
assert 'VIBEBOSS HQ' in ac
assert 'Phase:' in ac
assert 'Next:' in ac
print('OK:', len(ac), 'chars')
print(ac[:600])
"
```

## Phase 3 — Bug-fix (≥ 3 rounds)

Each round: run boot.sh → inspect JSON → fix any malformed output → re-run.

Known risk areas:
- Quotes in STATE.md content breaking JSON (mitigated: Python handles via env var)
- awk pattern mismatch if STATE.md heading format changes
- crew.yml parsing edge cases (null values, quoted dates)
- Empty directories causing `ls` to fail with set -e

## Phase 4 — Fresh-agent review

Dispatch Agent with: spec + settings.json content + boot.sh content. Ask for:
(1) broken logic, (2) spec gaps, (3) safety/edge-case issues.

## Phase 5 — Tighten (≥ 3 rounds)

- Round 1: Code clarity — rename opaque vars, remove redundancy
- Round 2: Test quality — add edge cases (missing files, empty runlog, malformed crew.yml)
- Round 3: Hardening — graceful degradation on every file read; ensure JSON is always valid
